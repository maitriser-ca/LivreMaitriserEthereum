[[installing_truffle]]
===== Installation du cadre de développement Truffle

(((&quot;Node.js&quot;)))(((&quot;Truffle&quot;,&quot;installation&quot;)))Le cadre de développement Truffle comprend plusieurs packages Node.js. Avant d&#39;installer +truffle+, vous devez avoir une installation à jour et fonctionnelle de Node.js et du Node Package Manager (+npm+).

La méthode recommandée pour installer Node.js et +npm+ consiste à utiliser le gestionnaire de version de nœud (+nvm+). Une fois que vous avez installé +nvm+, il gérera toutes les dépendances et mises à jour pour vous. Suivez les instructions trouvées sur http://nvm.sh[].

Une fois que +nvm+ est installé sur votre système d&#39;exploitation, l&#39;installation de Node.js est simple. Utilisez le drapeau +--lts+ pour dire à +nvm+ que vous voulez la version la plus récente de &quot;support à long terme&quot; (LTS) de Node.js :

++++
<pre data-type="programlisting">
$ <strong>nvm install --lts</strong>

</pre>
++++

Vérifiez que +node+ et +npm+ sont installés :

++++
<pre data-type="programlisting">
$ <strong>node -v</strong>
v8.9.4
$ <strong>npm -v</strong>
5.6.0

</pre>
++++

Ensuite, créez un fichier caché, _.nvmrc_, qui contient la version Node.js prise en charge par votre DApp afin que les développeurs n&#39;aient qu&#39;à exécuter &quot;nvm install&quot; à la racine du répertoire du projet et il s&#39;installera automatiquement et passera à l&#39;utilisation de cette version. :

++++
<pre data-type="programlisting">
$ <strong>node -v &gt; .nvmrc</strong>
$ <strong>nvm install</strong>

</pre>
++++

Ça parait bien. Maintenant, installons +truffle+ :

++++
<pre data-type="programlisting">
$ <strong>npm -g install truffle</strong>

+ truffle@4.0.6
installed 1 package in 37.508s

</pre>
++++

[[truffle_project_directory]]
===== Création d&#39;un répertoire de projet truffle

(((&quot;Truffle&quot;,&quot;création d&#39;un répertoire de projet&quot;, id=&quot;ix_appdx-dev-tools-asciidoc3&quot;, range=&quot;startofrange&quot;)))Pour chaque projet où vous utiliserez +truffle+, créez un répertoire de projet et initialisez +truffle+ dans ce répertoire. +truffle+ créera la structure de répertoire nécessaire dans le répertoire de votre projet. Il est d&#39;usage de donner au répertoire du projet un nom qui décrit le projet. Pour cet exemple, nous utiliserons +truffle+ pour déployer notre contrat +Faucet+ depuis &lt;<simple_contract_example>&gt;, et donc nous nommerons le dossier du projet __Faucet__ :

++++
<pre data-type="programlisting">
$ <strong>mkdir Faucet</strong>
$ <strong>cd Faucet</strong>
Faucet $

</pre>
++++

Une fois dans le répertoire _Faucet_, nous initialisons +truffle+ :

++++
<pre data-type="programlisting">
Faucet $ <strong>truffle init</strong>

</pre>
++++

+truffle+ crée une structure de répertoires et des fichiers par défaut :

----
Faucet
+---- contracts
|   `---- Migrations.sol
+---- migrations
|   `---- 1_initial_migration.js
+---- test
+---- truffle-config.js
`---- truffle.js
----

Nous utiliserons également un certain nombre de packages de support JavaScript (Node.js), en plus de +truffle+ lui-même. Nous pouvons les installer avec +npm+. Nous initialisons la structure de répertoires +npm+ et acceptons les valeurs par défaut suggérées par +npm+ :

++++
<pre data-type="programlisting">
$ <strong>npm init</strong>

package name: (faucet)
version: (1.0.0)
description:
entry point: (truffle-config.js)
test command:
git repository:
keywords:
author:
license: (ISC)
About to write to Faucet/package.json:

{
  "name": "faucet",
  "version": "1.0.0",
  "description": "",
  "main": "truffle-config.js",
  "directories": {
    "test": "test"
  },
  "scripts": {
    "test": "echo \"Error: no test specified\" &amp;&amp; exit 1"
  },
  "author": "",
  "license": "ISC"
}


Is this ok? (yes)

</pre>
++++

Maintenant, nous pouvons installer les dépendances que nous utiliserons pour faciliter le travail avec +truffle+ :

++++
<pre data-type="programlisting">
$ <strong>npm install dotenv truffle-wallet-provider ethereumjs-wallet</strong>

</pre>
++++

Nous avons maintenant un répertoire __node_modules__ avec plusieurs milliers de fichiers dans notre répertoire _Faucet_.

Avant de déployer un DApp dans un environnement de production cloud ou d&#39;intégration continue, il est important de spécifier le champ +engines+ afin que votre DApp soit construit avec la bonne version de Node.js et que ses dépendances associées soient installées. Pour plus de détails sur la configuration de ce champ, consultez la http://bit.ly/2zp2GPF[documentation].(((range=&quot;endofrange&quot;, startref=&quot;ix_appdx-dev-tools-asciidoc3&quot;)))

===== Configuration de truffle

(((&quot;Truffle&quot;,&quot;configuration&quot;)))+truffle+ crée des fichiers de configuration vides, _truffle.js_ et _truffle-config.js_. Sur les systèmes Windows, le nom _truffle.js_ peut provoquer un conflit lorsque vous essayez d&#39;exécuter la commande +truffle+ et que Windows tente d&#39;exécuter _truffle.js_ à la place. Pour éviter cela, nous allons supprimer _truffle.js_ et utiliser _truffle-config.js_ (en soutien aux utilisateurs de Windows, qui, honnêtement, souffrent déjà assez) :

++++
<pre data-type="programlisting">
$ <strong>rm truffle.js</strong>

</pre>
++++

Maintenant, nous éditons _truffle-config.js_ et remplaçons le contenu par l&#39;exemple de configuration montré ici :

[source,javascript]
----
module.exports = {
  networks: {
    localnode: { // Whatever network our local node connects to
      network_id: "*", // Match any network ID
      host: "localhost",
      port: 8545,
    }
  }
};
----

Cette configuration est un bon point de départ. Il configure un réseau Ethereum par défaut (nommé +localnode+), qui suppose que nous exécutons un client Ethereum tel que Parity, soit en tant que nœud complet, soit en tant que client léger. Cette configuration demandera à +truffle+ de communiquer avec le nœud local via RPC, sur le port 8545. +truffle+ utilisera le réseau Ethereum auquel le nœud local est connecté, comme le réseau principal Ethereum ou un réseau de test comme Ropsten. Le nœud local fournira également la fonctionnalité de portefeuille.

Dans les sections suivantes, nous configurerons des réseaux supplémentaires pour +truffle+ à utiliser, tels que la chaîne de blocs de test locale +ganache+ et Infura, un fournisseur de réseau hébergé. Au fur et à mesure que nous ajouterons des réseaux, le fichier de configuration deviendra plus complexe, mais il nous donnera également plus d&#39;options pour notre flux de travail de test et de développement.

===== Utiliser truffle pour déployer un contrat

(((&quot;contrats intelligents&quot;,&quot;utiliser Truffle pour déployer&quot;)))(((&quot;Truffle&quot;,&quot;déploiement de contrat avec&quot;)))Nous avons maintenant un répertoire de travail de base pour notre projet _Faucet_, et nous avons +truffle+ et ses dépendances configurées. Les contrats vont dans le sous-répertoire _contracts_ de notre projet. Le répertoire contient déjà un contrat &quot;d'aide&quot;, _Migrations.sol_, qui gère pour nous les mises à jour des contrats. Nous examinerons l&#39;utilisation de _Migrations.sol_ dans la section suivante.

Copions le contrat _Faucet.sol_ (depuis &lt;<solidity_faucet_example>&gt;) dans le sous-répertoire _contracts_, de sorte que le répertoire du projet ressemble à ceci :

----
Faucet
+---- contracts
|   +---- Faucet.sol
|   `---- Migrations.sol
...
----

Nous pouvons maintenant demander à +truffle+ de compiler le contrat pour nous :

++++
<pre data-type="programlisting">
$ <strong>truffle compile</strong>
Compiling ./contracts/Faucet.sol...
Compiling ./contracts/Migrations.sol...
Writing artifacts to ./build/contracts

</pre>
++++

[[truffle_migrations_understanding_deployment_scripts]]
===== Migrations Truffle—comprendre les scripts de déploiement

(((&quot;scripts de déploiement&quot;, id=&quot;ix_appdx-dev-tools-asciidoc4&quot;, range=&quot;startofrange&quot;)))(((&quot;migrations&quot;, id=&quot;ix_appdx-dev-tools-asciidoc5&quot;, range=&quot;startofrange&quot;)))(((&quot;Truffle&quot;,&quot;migrations&quot;, id=&quot;ix_appdx-dev-tools-asciidoc6&quot;, range=&quot;startofrange&quot;)))Truffle propose un système de déploiement appelé _migration_. Si vous avez travaillé dans d&#39;autres cadres de développement, vous avez peut-être vu quelque chose de similaire : Ruby on Rails, Python Django et de nombreux autres langages et cadres de développement ont une commande +migrate+.

Dans tous ces cadres de développement, le but d&#39;une migration est de gérer les modifications du schéma de données entre différentes versions du logiciel. Le but des migrations dans Ethereum est légèrement différent. Parce que les contrats Ethereum sont immuables et coûtent du gaz à déployer, Truffle propose un mécanisme de migration pour garder une trace des contrats (et des versions) qui ont déjà été déployés. Dans un projet complexe avec des dizaines de contrats et des dépendances complexes, vous ne voudriez pas avoir à payer pour redéployer des contrats qui n&#39;ont pas changé. Vous ne voudriez pas non plus suivre manuellement quelles versions de quels contrats ont déjà été déployées. Le mécanisme de migration Truffle fait tout cela en déployant le contrat intelligent _Migrations.sol_, qui suit ensuite tous les autres déploiements de contrat.

Nous n&#39;avons qu&#39;un seul contrat, _Faucet.sol_, ce qui signifie que le système de migration est pour le moins exagéré. Malheureusement, nous devons l&#39;utiliser. Mais, en apprenant à l&#39;utiliser pour un contrat, nous pouvons commencer à mettre en pratique de bonnes habitudes pour notre flux de travail de développement. L&#39;effort sera payant au fur et à mesure que les choses se compliqueront.

Le répertoire _migrations_ de Truffle est l&#39;endroit où se trouvent les scripts de migration. À l&#39;heure actuelle, il n&#39;y a qu&#39;un seul script, __1_initial_migration.js__, qui déploie le contrat _Migrations.sol_ lui-même :


[source,javascript,linenums]
----
include::code/truffle/Faucet/migrations/1_initial_migration.js[]
----

Nous avons besoin d&#39;un deuxième script de migration, pour déployer _Faucet.sol_. Appelons-le __2_deploy_contracts.js__. C&#39;est très simple, tout comme __1_initial_migration.js__, avec seulement quelques petites modifications. En fait, vous pouvez copier le contenu de __1_initial_migration.j__ et simplement remplacer toutes les instances de +Migrations+ par +Faucet+ :

[source,javascript,linenums]
----
include::code/truffle/Faucet/migrations/2_deploy_contracts.js[]
----

Le script initialise une variable +Faucet+, identifiant le code source Solidity _Faucet.sol_ comme l&#39;artefact qui définit +Faucet+. Ensuite, il appelle la fonction &quot;deploy&quot; pour déployer ce contrat.

Nous sommes prêts. Utilisons +truffle migrate+ pour le déployer. Nous devons spécifier sur quel réseau déployer le contrat, en utilisant l&#39;argument +--network+ . Nous n&#39;avons qu&#39;un seul réseau spécifié dans le fichier de configuration, que nous avons nommé +localnode+. Assurez-vous que votre client Ethereum local est en cours d&#39;exécution, puis tapez :

++++
<pre data-type="programlisting">
Faucet $ <strong>truffle migrate --network localnode</strong>

</pre>
++++

Comme nous utilisons un nœud local pour nous connecter au réseau Ethereum et gérer notre portefeuille, nous devons autoriser la transaction créée par +truffle+. Nous exécutons +parity+ connecté à la chaîne de blocs de test de Ropsten, donc pendant la migration, nous verrons une fenêtre contextuelle comme celle de &lt;<parity_deployment_confirmation>&gt; sur la console web de Parity.

===== Utilisation de la console Truffle

(((&quot;Truffle&quot;,&quot;console&quot;, id=&quot;ix_appdx-dev-tools-asciidoc7&quot;, range=&quot;startofrange&quot;)))Truffle propose une console JavaScript que nous pouvons utiliser pour interagir avec le réseau Ethereum (via le noeud local), interagir avec les contrats déployés et interagir avec le fournisseur de portefeuille. Dans notre configuration actuelle (+localnode+), le fournisseur de nœud et de portefeuille est notre client Parity local.

Démarrons la console Truffle et essayons quelques commandes :

++++
<pre data-type="programlisting">
$ <strong>truffle console --network localnode</strong>
truffle(localnode)>

</pre>
++++

Truffle présente une invite indiquant la configuration réseau sélectionnée (+localnode+).

[TIP]
====
Il est important de se rappeler et de savoir quel réseau vous utilisez. Vous ne voudriez pas déployer accidentellement un contrat de test ou effectuer une transaction sur le réseau principal Ethereum. Cela pourrait être une erreur coûteuse!
====

La console Truffle offre une fonction de saisie semi-automatique qui nous permet d&#39;explorer facilement l&#39;environnement. Si nous appuyons sur Tab après une commande partiellement terminée, Truffle terminera la commande pour nous. Appuyer deux fois sur Tab affichera toutes les complétions possibles si plus d&#39;une commande correspond à notre entrée. En fait, si nous appuyons deux fois sur Tab sur une invite vide, Truffle liste toutes les commandes disponibles :

La grande majorité des fonctions liées au portefeuille et aux nœuds sont fournies par l&#39;objet +web3+, qui est une instance de la bibliothèque web3.js. L&#39;objet +web3+ résume l&#39;interface RPC à notre nœud Parity. Vous remarquerez également deux objets aux noms familiers : +Migrations+ et +Faucet+. Ceux-ci représentent les contrats que nous venons de déployer. Nous utiliserons la console Truffle pour interagir avec un contrat. Tout d&#39;abord, vérifions notre wallet via l&#39;objet +web3+ :

++++

Notre client Parity a deux portefeuilles, avec un peu d&#39;ether de test sur Ropsten. L&#39;attribut +web3.eth.accounts+ contient une liste de tous les comptes. Nous pouvons vérifier le solde du premier compte en utilisant la fonction +getBalance+ :

++++
<pre data-type="programlisting">

truffle(localnode)&gt;

web3.js est une grande bibliothèque JavaScript qui offre une interface complète au système Ethereum, via un fournisseur tel qu&#39;un client local. Nous examinerons web3.js plus en détail dans &lt;<web3js_tutorial>&gt;. Essayons maintenant d&#39;interagir avec nos contrats :

++++
<pre data-type="programlisting">

truffle(localnode)&gt; <strong>web3.eth.getBalance(Faucet.address).toNumber()</strong>

Ensuite, nous utiliserons +sendTransaction+ pour envoyer de l&#39;ether de test afin de financer le contrat +Faucet+. Notez l&#39;utilisation de +web3.utils.toWei+ pour convertir les unités ether pour nous. Taper 18 zéros sans se tromper est à la fois difficile et dangereux, il est donc toujours préférable d&#39;utiliser un convertisseur d&#39;unités pour les valeurs. Voici comment nous envoyons la transaction :

++++
<pre data-type="programlisting">
truffle(localnode)&gt; <strong>web3.eth.sendTransaction({from:web3.eth.accounts[0],

==== Embark

(((&quot;Embark&quot;)))(((&quot;cadres de développement&quot;,&quot;Embark&quot;)))GitHub : https://github.com/embark-framework/embark/[]

Documents : https://embark.status.im/docs/[]

Référentiel de packages +npm+ : https://www.npmjs.com/package/embark[]

Embark est un framework conçu pour permettre aux développeurs de développer et de déployer facilement des applications décentralisées.
Embark s&#39;intègre à Ethereum, IPFS, Whisper et Swarm pour offrir les fonctionnalités suivantes :

* Déployer automatiquement les contrats et les rendre disponibles en code JS.
* Surveiller les changements et mettez à jour les contrats pour les redéployer si nécessaire.
* Gérer et interagir avec différentes chaînes (par exemple, testnet, local, mainnet).
* Gérer des systèmes complexes de contrats interdépendants.
* Stocker et récupérer des données, y compris le téléchargement et la récupération de fichiers hébergés dans IPFS.
* Faciliter le processus de déploiement de l&#39;application complète sur IPFS ou Swarm.
* Envoyer et recevoir des messages via Whisper.

Vous pouvez l&#39;installer avec +npm+ :

++++
<pre data-type="programlisting">
$ <strong>npm -g install embark</strong>

</pre>
++++

Documentation : https://openzeppelin.org/api/docs/open-zeppelin.html[]

https://openzeppelin.org/[OpenZeppelin] est un cadre ouvert de contrats intelligents réutilisables et sécurisés dans le langage Solidity.

Il est piloté par la communauté, dirigé par l&#39;équipe https://zeppelin.solutions/[Zeppelin], avec plus d&#39;une centaine de contributeurs externes. L&#39;objectif principal du cadre est la sécurité, obtenue en appliquant des modèles de sécurité de contrat standard et les meilleures pratiques, en s&#39;appuyant sur toute l&#39;expérience que les développeurs Zeppelin ont acquise en https://blog.zeppelin.solutions/tagged/security[auditant] un grand nombre de contrats, et grâce à des tests et des audits constants de la part de la communauté qui utilise le cadre de développement comme base pour leurs applications dans le monde réel.

Le cadre de développement OpenZeppelin est la solution la plus largement utilisée pour les contrats intelligents Ethereum. Le cadre de développement dispose actuellement d&#39;une vaste bibliothèque de contrats, y compris des implémentations de jetons ERC20 et ERC721, de nombreuses variantes de modèles de vente publique et des comportements simples que l&#39;on trouve couramment dans des contrats tels que &quot;Ownable&quot;, &quot;Pausable&quot; ou &quot;LimitBalance&quot;. Les contrats de ce référentiel fonctionnent dans certains cas comme des implémentations standard _de facto_.

Le cadre est sous licence MIT, et tous les contrats ont été conçus avec une approche modulaire pour garantir la facilité de réutilisation et d&#39;extension. Ce sont des blocs de construction propres et basiques, prêts à être utilisés dans votre prochain projet Ethereum. Configurons le cadre et construisons une simple vente publique à l&#39;aide des contrats OpenZeppelin, pour démontrer à quel point il est facile à utiliser. Cet exemple souligne également l&#39;importance de réutiliser des composants sécurisés au lieu de les écrire par vous-même.

Tout d&#39;abord, nous devrons installer la bibliothèque +openzeppelin-solidity+ dans notre espace de travail. La dernière version au moment d&#39;écrire ces lignes est la v1.9.0, nous allons donc l&#39;utiliser :

++++
<pre data-type="programlisting">
$ <strong>mkdir sample-crowdsale</strong>
$ <strong>cd sample-crowdsale</strong>
$ <strong>npm install openzeppelin-solidity@1.9.0</strong>
$ <strong>mkdir contracts</strong>

</pre>
++++

Au moment de la rédaction, OpenZeppelin comprend plusieurs contrats de jetons de base qui suivent les normes ERC20, ERC721 et ERC827, avec différentes caractéristiques d&#39;émission, de limites, d&#39;acquisition, de cycle de vie, etc.

Créons un jeton ERC20 monnayable, ce qui signifie que l&#39;approvisionnement initial commence à 0 et que de nouveaux jetons peuvent être créés par le propriétaire du jeton (dans notre cas, le pass:[<span class="keep-together">contrat</span>] de vente publique) et vendus aux acheteurs. Pour ce faire, nous allons créer un fichier _contracts/SampleToken.sol_ avec le contenu suivant :

[source,solidity]
----
include::code/OpenZeppelin/contracts/SampleToken.sol[]
----

OpenZeppelin fournit déjà un contrat +MintableToken+ que nous pouvons utiliser comme base pour notre jeton, nous ne définissons donc que les détails spécifiques à notre cas. Ensuite, faisons le contrat de vente publique. Tout comme avec les jetons, OpenZeppelin propose déjà une grande variété de saveurs de vente publique. Actuellement, vous trouverez des contrats pour divers scénarios impliquant la distribution, l&#39;émission, le prix et la validation. Donc, disons que vous voulez fixer un objectif pour votre vente publique et s&#39;il n&#39;est pas atteint au moment où la vente se termine, vous voulez rembourser tous vos investisseurs. Pour cela, vous pouvez utiliser le contrat http://bit.ly/2yHoh65[+RefundableCrowdsale+]. Ou peut-être souhaitez-vous définir une vente publique avec un prix croissant pour inciter les premiers acheteurs ; il y a un contrat http://bit.ly/2PtWOys[+IncreasingPriceCrowdsale+] juste pour ça. Vous pouvez également mettre fin à la vente publique lorsqu&#39;une quantité spécifiée d&#39;ether a été reçue par le contrat (http://bit.ly/2OVsCN8[+CappedCrowdsale+]), ou définir une heure de fin avec le contrat http://bit.ly/2zp2Nuz[+TimedCrowdsale+], ou créez une liste blanche d&#39;acheteurs avec le contrat http://bit.ly/2CN8Hc9[+WhitelistedCrowdsale+].

Comme nous l&#39;avons dit précédemment, les contrats OpenZeppelin sont des éléments de base. Ces contrats de crowdsale ont été conçus pour être combinés ; il suffit de lire le code source du contrat de base http://bit.ly/2ABIQSI[+Crowdsale+] pour savoir comment l&#39;étendre. Pour la vente publique de notre jeton, nous devons créer des jetons lorsque l&#39;ether est reçu par le contrat de vente publique, utilisons donc http://bit.ly/2Sx3HOc[+MintedCrowdsale+] comme base. Et pour le rendre plus intéressant, faisons-en également une http://bit.ly/2Qef0Jm[+PostDeliveryCrowdsale+] afin que les jetons ne puissent être retirés qu&#39;après la fin de la vente publique. Pour ce faire, nous allons écrire ce qui suit dans _contracts/SampleCrowdsale.sol_ :

[source,solidity]
----
include::code/OpenZeppelin/contracts/SampleCrowdsale.sol[]
----

Encore une fois, nous avons à peine eu à écrire du code ; nous venons de réutiliser le code testé au combat que la communauté OpenZeppelin a mis à disposition. Cependant, il est important de noter que ce cas est différent de celui de notre contrat `SampleToken`. Si vous allez sur les http://bit.ly/2Q8lQ3o[Tests automatisés Crowdsale], vous verrez qu&#39;ils sont testés de manière isolée. Lorsque vous intégrez différentes unités de code dans un composant plus gros, il ne suffit pas de tester toutes les unités séparément, car les interactions entre elles peuvent provoquer des comportements auxquels vous ne vous attendiez pas. En particulier, vous verrez qu&#39;ici nous avons introduit l&#39;héritage multiple, ce qui peut surprendre le développeur s&#39;il ne comprend pas les détails de Solidity. Notre pass:[<span class="keep-together"><code>SampleCrowdsale</code></span>] est simple et fonctionnera exactement comme prévu, car le cadre a été conçu pour simplifier des cas comme ceux-ci ; mais ne relâchez pas votre vigilance à cause de la simplicité qu&#39;introduit ce cadre de développement. Chaque fois que vous intégrez des parties du cadre de développement OpenZeppelin pour créer une solution plus complexe, vous devez tester entièrement chaque aspect de votre solution pour vous assurer que toutes les interactions des unités fonctionnent comme vous le souhaitez.

Enfin, lorsque nous sommes satisfaits de notre solution et que nous l&#39;avons testée de manière approfondie, nous devons la déployer. ==== ZeppelinOS

(((&quot;cadres de développement&quot;,&quot;ZeppelinOS&quot;)))(((&quot;ZeppelinOS&quot;)))GitHhub : https://github.com/zeppelinos[]

Site Web : https://zeppelinos.org[]

Blogue : https://blog.zeppelinos.org[]

https://github.com/zeppelinos[ZeppelinOS] est &quot;une plate-forme open source distribuée d&#39;outils et de services au-dessus de l&#39;EVM
pour développer et gérer en toute sécurité des applications de contrats intelligents.

Contrairement au code d&#39;OpenZeppelin, qui doit être redéployé avec chaque application à chaque utilisation, le code de ZeppelinOS vit en chaîne. Les applications qui ont besoin d&#39;une fonctionnalité donnée, par exemple un jeton ERC20, non seulement n&#39;ont pas à reconcevoir et à réauditer sa mise en œuvre (quelque chose qu&#39;OpenZeppelin a résolu), mais n&#39;ont même pas besoin de la déployer. Avec ZeppelinOS, une application interagit directement avec l&#39;implémentation en chaîne du jeton, de la même manière qu&#39;une application de bureau interagit avec les composants de son système d&#39;exploitation sous-jacent.

(((&quot;proxy&quot;)))Au cœur de ZeppelinOS se trouve un contrat très intelligent connu sous le nom de _proxy_. Un proxy est un contrat capable d&#39;encapsuler n&#39;importe quel autre contrat, d&#39;exposer son interface sans avoir à implémenter manuellement des setters et des getters pour celui-ci, et de le mettre à niveau sans perdre son état. En termes de solidité, il peut être considéré comme un contrat normal dont la logique métier est contenue dans une bibliothèque, qui peut être échangée contre une nouvelle bibliothèque à tout moment sans perdre son état. La manière dont le proxy est lié à son implémentation est entièrement automatisée et encapsulée pour le développeur. Pratiquement n&#39;importe quel contrat peut être mis à niveau avec peu ou pas de changement dans son code. Plus d&#39;informations sur le mécanisme de proxy de ZeppelinOS peuvent être trouvées dans le http://bit.ly/2OfuNpu[blog], et un exemple de la façon de l&#39;utiliser peut être trouvé http://bit.ly/2OfuE5q[sur GitHub].

Le développement d&#39;applications à l&#39;aide de ZeppelinOS est similaire au développement d&#39;applications JavaScript à l&#39;aide de +npm+. Un +AppManager+ gère un package d&#39;application pour chaque version de l&#39;application. Un package est simplement un répertoire de contrats, chacun pouvant avoir un ou plusieurs proxys pouvant être mis à niveau. Le +AppManager+ fournit non seulement des proxys pour les contrats spécifiques à l&#39;application, mais le fait également pour les implémentations de ZeppelinOS, sous la forme d&#39;une bibliothèque standard. Pour en voir un exemple complet, veuillez visiter http://bit.ly/2PtyJb3[examples/complex].

Bien qu&#39;actuellement en développement, ZeppelinOS vise à fournir un large éventail de fonctionnalités supplémentaires, telles que des outils de développement, un planificateur qui automatise les opérations en arrière-plan dans les contrats, des primes de développement, un marché qui facilite la communication et l&#39;échange de valeur entre les applications, et bien plus encore. Tout cela est décrit dans le http://bit.ly/2QcxV7K[whitepaper].(((range=&quot;endofrange&quot;, startref=&quot;ix_appdx-dev-tools-asciidoc0&quot;))) de ZeppelinOS

=== Tester les contrats intelligents

(((&quot;cadres de développement&quot;,&quot;pour le développement de contrats intelligents&quot;, id=&quot;ix_appdx-dev-tools-asciidoc11&quot;, range=&quot;startofrange&quot;)))(((&quot;contrat intelligent&quot;,&quot;cadres de développement test&quot;, id=&quot; ix_appdx-dev-tools-asciidoc12&quot;, range=&quot;startofrange&quot;)))(((&quot;cadres de développement de test&quot;,&quot;pour le développement de contrats intelligents&quot;, id=&quot;ix_appdx-dev-tools-asciidoc13&quot;, range=&quot;startofrange&quot;) ))Il existe plusieurs cadres de test couramment utilisés pour le développement de contrats intelligents, résumés dans &lt;<testing_frameworks_table>&gt; :

[[testing_frameworks_table]]
[options="header"]
.Résumé des cadres de test des contrats intelligents
|=======
|Cadre | Langue(s) d&#39;essai | Cadre de test | Émulateur de chaîne | Site Internet
|Truffe | JavaScript/Solidity | Moka | TestRPC/Ganache | https://truffeframework.com/[]
|Embark | JavaScript | Moka | TestRPC/Ganache | https://embark.status.im/docs/[]

Truffle:: (((&quot;Truffle&quot;,&quot;comme cadre de développement de test&quot;)))Truffle permet d&#39;écrire des tests unitaires en JavaScript (basé sur Mocha) ou Solidity. Ces tests sont exécutés contre Ganache.

Embark:: (((&quot;Embark&quot;)))Embark s&#39;intègre à Mocha pour exécuter des tests unitaires écrits en JavaScript. Les tests sont à leur tour exécutés sur des contrats déployés sur TestRPC/Ganache. Le framework Embark déploie automatiquement les contrats intelligents et redéploie automatiquement les contrats lorsqu&#39;ils sont modifiés. Il assure également le suivi des contrats déployés et ne déploie les contrats que lorsque cela est vraiment nécessaire. Embark inclut une bibliothèque de test pour exécuter et tester rapidement vos contrats dans un EVM, avec des fonctions comme +assert.equal+. La commande +embark test+ exécutera tous les fichiers de test sous le répertoire _test_.

Dapp:: (((&quot;Dapp&quot;)))Dapp utilise le code Solidity natif (une bibliothèque appelée +ds-test+) et une bibliothèque Rust construite par Parity appelée +ethrun+ pour exécuter le code intermédiaire Ethereum, puis affirmer l&#39;exactitude. La bibliothèque +ds-test+ fournit des fonctions d&#39;assertion pour valider l&#39;exactitude et les événements pour la journalisation des données dans la console.
+
Les fonctions d&#39;assertion incluent :
+
----
assert(bool condition)
assertEq(address a, address b)
assertEq(bytes32 a, bytes32 b)
assertEq(int a, int b)
assertEq(uint a, uint b)
assertEq0(bytes a, bytes b)
expectEventsExact(address target)
----
+
Les commandes de journalisation consigneront les informations dans la console, ce qui les rendra utiles pour le débogage :
+
----
logs(bytes)
log_bytes32(bytes32)
log_named_bytes32(bytes32 key, bytes32 val)
log_named_address(bytes32 key, address val)
log_named_int(bytes32 key, int val)
log_named_uint(bytes32 key, uint val)
log_named_decimal_int(bytes32 key, int val, uint decimals)
log_named_decimal_uint(bytes32 key, uint val, uint decimals)
----

Populus:: (((&quot;Populus&quot;)))Populus utilise Python et son propre émulateur de chaîne pour exécuter des contrats écrits en Solidity. Les tests unitaires sont écrits en Python avec la bibliothèque +pytest+. Populus prend en charge la rédaction de contrats spécifiquement pour les tests. Ces noms de fichiers de contrat doivent correspondre au modèle global _Test*.sol_ et se trouver n&#39;importe où dans le répertoire des tests du projet, _tests_.

==== Test embarqué sur la chaîne de blocs

(((&quot;chaîne de blocs&quot;,&quot;test embarqué sur&quot;)))(((&quot;cadres de développement de test&quot;,&quot;tests embarqués sur&quot;)))Bien que la plupart des tests ne devraient pas avoir lieu sur les contrats déployés, le comportement d&#39;un contrat peut être vérifié via les clients Ethereum. Les commandes suivantes peuvent être utilisées pour évaluer l&#39;état d&#39;un contrat intelligent. Ces commandes doivent être tapées sur le terminal +geth+, bien que n&#39;importe quelle console web3 les prenne également en charge.

Pour obtenir l&#39;adresse d&#39;un contrat à __++txhash++__, utilisez :

++++
<pre data-type="programlisting">
web3.eth.getTransactionReceipt(<em>txhash</em>);

</pre>
++++


Cette commande obtient le code d&#39;un contrat déployé à __++contractaddress++__ ; cela peut être utilisé pour vérifier le bon déploiement :

++++
<pre data-type="programlisting">
web3.eth.getCode(<em>contractaddress</em>)

</pre>
++++

Cela permet d&#39;obtenir les journaux complets du contrat situé à l&#39;adresse spécifiée dans __++options++__, ce qui est utile pour afficher l&#39;historique des appels d&#39;un contrat :

++++
<pre data-type="programlisting">
web3.eth.getPastLogs(<em>options</em>)

</pre>
++++

Enfin, cette commande obtient le stockage situé à __++address++__ avec un décalage de __++position++__ :

++++
<pre data-type="programlisting">
web3.eth.getStorageAt(<em>address</em>, <em>position</em>)

</pre>
++++

==== Ganache : une chaîne de blocs de test locale

(((&quot;Ganache&quot;,&quot;test de chaîne de blocs local avec&quot;)))Ganache est une chaîne de blocs de test locale que vous pouvez utiliser pour déployer des contrats, développer vos applications et exécuter des tests. Il est disponible en tant qu&#39;application de bureau (avec une interface utilisateur graphique) pour Windows, macOS et Linux. Il est également disponible sous la forme d&#39;un utilitaire de ligne de commande appelé +ganache-cli+. Pour plus de détails et des instructions d&#39;installation pour l&#39;application de bureau Ganache, voir https://truffleframework.com/ganache[].

Le code +ganache-cli+ peut être trouvé sur https://github.com/trufflesuite/ganache-cli/[].

Pour installer la ligne de commande +ganache-cli+, utilisez +npm+ :

++++
<pre data-type="programlisting">
$ <strong>npm install -g ganache-cli</strong>

</pre>
++++

Vous pouvez utiliser +ganache-cli+ pour démarrer une chaîne de blocs locale à tester comme suit :

++++
<pre data-type="programlisting">
$ <strong>ganache-cli \
  --networkId=3 \
  --port="8545" \
  --verbose \
  --gasLimit=8000000 \
  --gasPrice=4000000000;</strong>
  
</pre>
++++

Quelques notes sur cette ligne de commande :

* [ ] Vérifiez que les valeurs des indicateurs `--networkId` et `--port` correspondent à votre configuration dans _truffle.js_.
* [ ] Vérifiez que la valeur de l&#39;indicateur `--gasLimit` correspond à la dernière limite de gaz du réseau principal (par exemple, 8 000 000 de gaz) indiquée sur https://ethstats.net pour éviter de rencontrer inutilement des exceptions «hors de gaz». Notez qu&#39;un `--gasPrice` de +4000000000+ représente un prix du gaz de 4 gwei.
